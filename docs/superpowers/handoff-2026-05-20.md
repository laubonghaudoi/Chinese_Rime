## 3. Phase 1 pitfalls вҖ” things that bit me and you should know

These are real-world surprises I hit during P1.T10. The fix-forward commits are now in the branch; the **plan has been amended to match** where applicable (see В§4). Listing them here so you don't repeat the discovery cost.

### 3.1 Upstream `*.schema.yaml` files use tabs in YAML-illegal positions

Roughly 30 of the 153 upstream schema files have:
- A trailing **tab** after a quoted value (e.g. `version: "2023.07.141"\t\n`)
- A **tab** used to align an inline `# comment` (e.g. `- xform/y([AQZ])/Y$1/\t#е№Ҫйҹ»дёҚеҲҶAB`)

`pyyaml.safe_load` rejects both with `ScannerError: found character '\t' that cannot start any token`. Per `AGENT.md` we do NOT modify upstream files. **Fix:** convert all tabs to spaces before parsing вҖ” safe because YAML forbids tabs as indentation, so any tab in these files is in post-content position. Implemented at `script/manifest/walk_submodules.py:_load_one`.

### 3.2 One schema's `description` is a YAML list, not a string

`sources/жҷүиӘһ/жҷӢиҜӯжұҫйҳіиҜқиҫ“е…Ҙж–№жЎҲ/fengyo.schema.yaml` uses `description:` followed by a YAML sequence of strings. The naive `.strip()` on the value blows up with `AttributeError: 'list' object has no attribute 'strip'`. **Fix:** `_normalize_description` helper in `walk_submodules.py` joins list-shaped descriptions with newlines, then `.strip()`s.

### 3.3 The README's `<code>schema_id</code>` labels don't always match the actual `schema_id:` field in the YAML

Four case/spelling mismatches I had to fix in `script/.readme_ids.txt` and `script/manifest_overrides.yaml`:

| README label | Actual schema_id |
|---|---|
| `OC_NULLL` | `OC_nulll` |
| `Dangrvond` | `dangrvond` |
| `qyeyshanglr_hanja` | `qyeishanglr_hanja` |
| `jyutping+` | `jyutping_plus` |
| `jutjnyu_gaulaupin_dangjunbikwaa` | `gaulau_jyutping_dangjun` |

**Lesson:** the YAML's `schema_id` field is the canonical truth. The README's `<code>...</code>` blocks are display labels and may drift. When you encounter a similar mismatch later (e.g. Phase 7 prose), trust the manifest.

### 3.4 Four README-listed schemas exist only in the README

`BekIngLMC` (зҰҸе·һи©ұе…«йҹізҫ…йҰ¬еӯ—), `GauHuoiLMC` (зҰҸе·һи©ұж•ҷжңғзҫ…йҰ¬еӯ—), `OC_schuesslerLHC` (жқұжјўжјўиӘһ иЁұжҖқиҗҠ), `jutjnyu_gaulaupin_dangjunbikwaa` (the long-form variant) вҖ” listed in `README.md` but no matching `*.schema.yaml` exists in the submodules. They're now under the `orphans:` block of `manifest_overrides.yaml`. The site will render them as no-recipe entries.

### 3.5 Some recipes ship variant schemas under nested subdirs

`sources/иҘҝеҚ—е®ҳи©ұ/иңҖжӢј/mobile/shupin_*.schema.yaml` are mobile-keyboard variants with the **same `schema_id` as their parent**. A naive `rglob("*.schema.yaml")` plus dict-keyed-by-`schema_id` silently drops one of each pair. Meanwhile `sources/е®ўе®¶и©ұ/е®ўжӢј/hakka-dict-rime/hakka_pinyin.schema.yaml` is legitimately 4-deep with a unique schema_id, so you can't just limit walk depth.

**Fix:** walk recursively but sort by `(len(path.parts), path)` and de-duplicate by schema_id, keeping the shallowest. Implemented in `walk_submodules.walk_submodules` and `_iter_schema_yamls`.

### 3.6 Subagent tooling drift: `uv` vs `pip`

Some implementer subagents used `uv` to manage the venv (rather than the plan's plain `pip install -e ".[dev]"`). This produced a stray `script/uv.lock` and a different lockfile philosophy. Now ignored in `script/.gitignore`. **Tell GPT explicitly: use `pip` + `python -m venv`, not `uv`.** The plan's wording says pip; subagents should honour it.

## 7. Recommended pacing for GPT Codex

In my Phase 1 experience, the per-task pattern was:
1. Dispatch implementer subagent with full task text (model: haiku for mechanical, sonnet for transcription)
2. Run spec compliance reviewer subagent (haiku)
3. Run code quality reviewer subagent (haiku, model: code-reviewer)
4. Apply small fix-forwards in the controller

That's 3+ subagents per task Г— 43 tasks = ~130+ dispatches if you go strictly by-the-book. I shortcut this after the pattern was established (verified manually for trivial TDD tasks). Apply your own judgment:

- **TDD tasks with verbatim plan code** (Phase 1.3вҖ“1.7 pattern): implementer + manual verification is enough.
- **Astro component / page tasks** (most of Phases 2вҖ“7): implementer + `pnpm check` + visual verification in dev server.
- **Integration/workflow tasks** (Phase 8): implementer + reviewer + manual workflow dispatch.

If you stick to the full two-stage review, this is a multi-day workstream. If you shortcut where safe, it's still 1вҖ“2 long sessions.

## 8. Spec/plan details GPT should pay close attention to

These are the spots in the plan/spec where it'd be easy to misread:

- **Visual system:** spec В§6 defines color tokens (`--bg-page: #cfcfcf`, `--zhu: #b8281f`, etc.), type scale, and component inventory. Phase 3 components must use these tokens via `var(--token-name)`, not hardcoded hex. The high-contrast direction (no light-gray borders) was the user's explicit choice вҖ” see brainstorming history.
- **All counts in the spec are illustrative.** The site renders numbers from `schemas.json` at build time. Don't hardcode `144 еҖӢж–№жЎҲ` etc. in components.
- **The еҚ° logo is swappable.** `site/public/brand/logo.svg` is a single asset; the `Brand`/`Chop` components read it. Replacing the chop later = one file change.
- **Schema URLs use `slug`, not `schema_id`.** `slug` is the URL-safe form. Already in the manifest (e.g. `jyutping_plus.slug === "jyutping_plus"`, since the rename happened pre-emit). For new edge cases, `_slugify()` does `+` вҶ’ `-plus`.
- **Per-schema page has 12 sections, including 4 separate `йҹізі»иіҮж–ҷ В· 01вҖ“04` placeholder sections** for иҒІжҜҚиЎЁ / йҹ»жҜҚиЎЁ / иҒІиӘҝ / еӯ—йҹіе°Қз…§. These ship in v1 as layout placeholders; the actual data extraction is the largest single v2 effort. Reference: [jyutping.org/jyutping/](https://jyutping.org/jyutping/).
- **zh-Hant only for v1, no English** вҖ” but reserve i18n hooks. Don't add an English route or toggle; just don't paint the architecture into a zh-only corner.
- **`/catalog/` dialect anchors use Chinese-character heading IDs** (auto-generated by Astro). URL-encoded in the bar but functional. No `dialect_slugs.yaml` curation.
